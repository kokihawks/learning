# DDD知識の優先度マップ

## このドキュメントの目的

DDDは概念が多く、学習者が「何から手をつけるべきか」で迷いやすい。
本ドキュメントでは、DDD理解に必要な知識を **4段階の優先度** に分類し、
学習順序の指針を示す。

DDDの全体像をつかみたい場合は、まず [DDDについて](DDDについて.md) を読んでから本マップに戻ってくることを推奨する。

## 優先度の凡例

| 優先度 | 意味 |
|--------|------|
| 🔴 **S** | 最重要（これを知らずしてDDDは理解できない） |
| 🟠 **A** | 重要（実装に踏み込むなら必須） |
| 🟡 **B** | 応用（複雑な設計判断に必要） |
| 🟢 **C** | 発展（DDDを使いこなす段階） |

## 推奨学習順序

基本は **S → A → B → C** の順で進める。各層の中の順序は以下を推奨:

### S層（戦略的設計）の推奨順序
1. **ドメインとサブドメイン** - 全体像と「どこに価値があるか」を掴む
2. **境界づけられたコンテキスト** - システムをどう区切るかを学ぶ
3. **ユビキタス言語** - コンテキスト内で成立する共通言語を理解
4. **ドメインモデル** - モデル＝コードへの橋渡しを掴む

### A層（戦術的設計）の推奨順序
1. **エンティティと値オブジェクト** - モデリングの最小単位の区別を掴む
2. **集約と集約ルート** - Entity を業務的にまとまった単位へ束ねる
3. **リポジトリ** - 集約の永続化を抽象化する
4. **ドメインサービス** - Entity/VO に収まらないロジックの置き場
5. **レイヤードアーキテクチャ** - ここまでの要素を層構造に配置する

### 学習の進め方のコツ
- S層は「思想・概念」が中心。コードを書く前に**腹落ちさせる**ことを優先
- A層は実装パターン。具体例を写経しながら手を動かす
- B層・C層は **必要になった時に都度参照** すれば良い
