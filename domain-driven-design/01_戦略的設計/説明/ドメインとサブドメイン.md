# ドメインとサブドメイン

## 1. 定義（〜とは）

### ドメイン
**ドメイン（Domain）** とは、ソフトウェアが解決しようとしているビジネス領域・知識領域全体を指す。会社や組織が事業として扱っている領域そのものと言える。

### サブドメイン
**サブドメイン（Subdomain）** とは、大きなドメインを意味のある単位に分割した小領域。DDDではサブドメインを以下の3種類に分類する:

| 種類 | 説明 | 競争優位への寄与 |
|------|------|----------------|
| **コアドメイン**（Core Subdomain） | ビジネスの差別化要因となる最重要領域 | ⭐⭐⭐ 大 |
| **支援サブドメイン**（Supporting Subdomain） | コアを支えるが、それ自体は差別化要因ではない | ⭐⭐ 中 |
| **汎用サブドメイン**（Generic Subdomain） | どの会社でも共通の汎用機能 | ⭐ 小 |

> 一言メタファー: **コア=必殺技 / 支援=必殺技を成立させる準備 / 汎用=どこにでもある日用品**

## 2. なぜ必要か / 解決する問題

### サブドメイン分類がない場合に起きる問題
- **全機能を均等に扱ってしまう**: 差別化要因の機能と、どこにでもある機能（会計、認証など）を同じ熱量で作ってしまい、コアの品質が上がらない
- **「内製 vs 外部利用」の判断軸がない**: 本来SaaSや既製品で済ませるべき領域に時間を浪費する
- **チームのリソース配分が誤る**: 最も価値を生む領域に最も優秀な人材を配置できない

### サブドメイン分類が解決すること
- **力の入れどころが明確になる**: コアドメインに最も投資し、汎用サブドメインは外部サービスや既製品で済ませる判断ができる
- **DDDの戦術的パターン適用範囲を絞れる**: コアドメインにこそDDDの全戦術パターン（集約、ドメインサービス等）を適用し、汎用領域では軽量に作る

## 3. 具体例

### 例①: Eコマース
- **コアドメイン**: レコメンドエンジン（ユーザー体験の差別化要因）/ 注文管理（独自のビジネスルール）
- **支援サブドメイン**: 在庫管理 / 配送追跡
- **汎用サブドメイン**: 認証認可 / 決済処理 / 会計

> Eコマース企業にとって「決済」は重要だが、Stripe等の外部サービスを使うのが普通。一方「注文ルール」は自社の競争力に直結するため、丁寧にモデリングすべき。

### 例②: 物流会社
- **コアドメイン**: 配送ルート最適化アルゴリズム（コスト・速度の差別化要因）
- **支援サブドメイン**: 倉庫の在庫管理 / ドライバー管理
- **汎用サブドメイン**: 給与計算 / 経費精算

> 物流会社にとってルート最適化は他社との差別化点。給与計算は外部の人事SaaSで十分。

## 4. よくある誤解・アンチパターン

### 誤解①: 「機能の重要度＝コアドメイン」
**誤り**: 「ログイン機能はないと困るからコアドメインだ」
**正しい考え方**: ログインは「ないと困る」が、**他社との差別化要因にはならない** ので汎用サブドメイン。コアドメインは「他社との競争で勝つための領域」。

### 誤解②: 「サブドメイン＝マイクロサービスの単位」
**誤り**: 「サブドメインに分けたから、そのままマイクロサービスにしよう」
**正しい考え方**: サブドメインは **ビジネス上の関心の分割**。マイクロサービスは **技術的な実装の分割**。両者は近いが必ずしも一致しない（→ 詳細は「境界づけられたコンテキスト」へ）。

### 誤解③: 「汎用サブドメインは雑に作って良い」
**誤り**: 「会計部分は汎用だから雑なコードでOK」
**正しい考え方**: 「雑に作る」のではなく「**そもそも自社で作らず外部利用を検討する**」が正解。作らざるを得ないなら、シンプルに作って良い、という意味。

### アンチパターン: コアドメインへの投資不足
全領域を均等に作った結果、コアドメインに使える時間が減り、競争力のない凡庸なシステムになる。
