````markdown
# エンティティと値オブジェクト

## 1. 定義（〜とは）

### エンティティ
**エンティティ（Entity）** とは、ライフサイクルを通じて **一意な同一性（identity）** を持つオブジェクト。属性が変わっても「同じ〇〇である」と追跡され続ける。識別は ID で行う。

### 値オブジェクト
**値オブジェクト（Value Object, VO）** とは、**属性の組そのもの** を表す不変オブジェクト。属性が全て等しければ交換可能とみなす。ID を持たず、同一性ではなく等価性で比較する。

DDD 原典では、両者は戦術的設計の最も基礎的なビルディングブロックとして位置づけられる（Evans, DDD chapter 5）。

> 一言メタファー: **エンティティ=人**（名前や住所が変わっても同じ人）、**値オブジェクト=お金**（1000円札と1000円札は交換可能）

## 2. なぜ必要か / 解決する問題

### 区別がない場合に起きる問題
- **全てをエンティティで扱う**: 住所や金額にまで ID を付け、同じ内容なのに別物扱いされる。無駄なライフサイクル管理が発生し、ドメインが複雑化する
- **全てをただのデータ構造で扱う**: 不変性が保証されず、副作用が読めない。型が `string` や `number` ばかりになり、通貨違いや住所の不整合が実行時まで検知できない（Primitive Obsession）
- **同一性の基準が曖昧**: 「同じ顧客か」を判定する基準がコードごとに違う（名前一致 / ID一致 / メール一致）

### 区別が解決すること
- **同一性 vs 等価性の意図がコードで宣言される**: 何を以て "同じ" とするかが明確
- **VO は不変でスレッドセーフ**: 副作用が減り、計算結果を合成しやすい
- **エンティティのライフサイクル管理が一点集中**: 追跡対象が明確になる

## 3. 具体例

### 例①: Eコマース（住所 vs 顧客）

**悪い例: すべてをエンティティで扱う**

```typescript
class Address {
  constructor(
    public id: string,
    public postalCode: string,
    public city: string,
    public street: string,
  ) {}
}
```
→ 同じ「東京都渋谷区〇〇 1-2-3」でも ID が違えば別物扱い。顧客Aと顧客Bが同じ住所を共有しているか検知できず、重複判定も効かない。

**良い例: 住所は VO、顧客は Entity**

```typescript
class Address {  // 値オブジェクト
  constructor(
    readonly postalCode: string,
    readonly city: string,
    readonly street: string,
  ) {}

  equals(other: Address): boolean {
    return this.postalCode === other.postalCode
      && this.city === other.city
      && this.street === other.street;
  }
}

class Customer {  // エンティティ
  constructor(
    readonly id: CustomerId,
    private name: string,
    private address: Address,
  ) {}

  moveTo(newAddress: Address): void {
    this.address = newAddress;  // Address は丸ごと差し替え（不変）
  }

  equals(other: Customer): boolean {
    return this.id.equals(other.id);  // 同一性は ID で判定
  }
}
```

> ポイント: **住所は属性の束であり、内容が等しければ "同じ住所" と扱える**。顧客は ID で追跡され、引っ越し（属性変更）しても同じ顧客として残る。

### 例②: 金額（Money）— 典型的な VO

```typescript
class Money {
  private constructor(
    readonly amount: number,
    readonly currency: string,
  ) {}

  static yen(amount: number): Money { return new Money(amount, "JPY"); }
  static zero(): Money { return new Money(0, "JPY"); }

  plus(other: Money): Money {
    if (this.currency !== other.currency) {
      throw new Error("currency mismatch");
    }
    return new Money(this.amount + other.amount, this.currency);
  }

  equals(other: Money): boolean {
    return this.amount === other.amount && this.currency === other.currency;
  }
}
```

> ポイント: `Money.yen(1000).equals(Money.yen(1000))` は `true`。計算操作は必ず新しい `Money` を返す（不変）。`number` 単独では表現できない「通貨の一致」をコンパイル時に強制できる。

## 4. よくある誤解・アンチパターン

### 誤解①: 「ID があればエンティティ、なければ VO」
**誤り**: 「このクラスには `id` フィールドがあるからエンティティだ」
**正しい考え方**: ID の有無ではなく、**同一性をどう扱うか** が本質。DB のサロゲートキーのために ID を付けただけのクラスは VO として扱うべき場合がある。

### 誤解②: 「金額や日付はプリミティブで十分」
**誤り**: 「`Money` なんて作らず `number` で良い」
**正しい考え方**: 通貨単位や小数誤差、日付範囲の制約などは、プリミティブのままでは表現できない。VO にすることで **不正な状態を型レベルで排除** できる（Primitive Obsession の回避）。

### 誤解③: 「VO は `readonly` を付ければ完成」
**誤り**: 「フィールドに `readonly` を付ければ VO だ」
**正しい考え方**: 不変性に加え、**等価性の判定基準**（`equals()`）と **操作の結果が新しい VO を返す** という2点が揃って VO が成立する。setter を生やした瞬間 VO ではなくなる。

### アンチパターン: Primitive Obsession（原始型中毒）
ドメインの意味を持つ値（金額・住所・識別子など）を `string` や `number` のまま扱う。型の手がかりがなくなり、「商品IDと顧客IDを取り違える」「円とドルを足す」といったバグを実行時まで検知できない。

## 5. 実装上の判断軸

### エンティティを採用すべきサイン
- 同じ属性でも別物として追跡する必要がある（顧客・注文・口座）
- ライフサイクルがある（作成 → 変更 → 削除、状態遷移）
- 履歴や監査の対象になる
- 属性が時間とともに変化しても「同じもの」として扱いたい

### 値オブジェクトを採用すべきサイン
- 属性が等しければ完全に交換可能（金額・住所・日付範囲・座標）
- 不変であるべき（一度決めたら変わらない）
- ID を持たせる意味がない
- 計算や合成の結果として新しい値を作りたい

### 判断に迷ったときのチェックリスト
- [ ] このオブジェクトの 2 インスタンスは、属性が同じなら "同じ" と扱えるか？ → Yes なら VO

- [ ] 履歴やライフサイクルを追跡する必要があるか？ → Yes なら Entity
- [ ] この値を「合成」「加算」「結合」したくなるか？ → Yes なら VO
````

````markdown
# 集約と集約ルート

## 1. 定義（〜とは）

**集約（Aggregate）** とは、一貫性を保つべき Entity と VO のまとまり。**集約ルート（Aggregate Root）** は集約の唯一の入口となる Entity で、外部からの操作は必ずルートを経由する。集約内部の要素には、外部から直接アクセスしない。

原典: Evans, DDD chapter 6。DDD の戦術的パターンの中でも、整合性設計の中心的概念。

> 一言メタファー: **家族のまとまり**。外部から見れば「〇〇家」という単位で契約や郵便物が届く。家族メンバー個人に外から直接口出ししない。

## 2. なぜ必要か / 解決する問題

### 集約がない場合に起きる問題
- **不変条件が崩れる**: 「注文合計は明細の合計と一致する」「注文明細は 1 つ以上必要」などのルールを複数箇所で守ろうとして、結局どこかで破られる
- **ロック粒度の設計指針がない**: 何をまとめてロックするかが不明瞭で、デッドロックや不整合が起きる
- **並行更新時の衝突**: 複数リクエストが別 Entity を独立に更新し、全体としては矛盾した状態が残る
- **永続化単位が曖昧**: 関連する Entity 群をバラバラに保存して、中途半端な状態が DB に残る

### 集約が解決すること
- **不変条件がルート内でチェックされる**: 集約の内部を守る責務がルートに集中
- **トランザクション境界と一致**: 1 トランザクション = 1 集約 の原則で、ACID を効かせやすい
- **並行制御の単位が明確**: 楽観ロックや悲観ロックを集約単位でかける
- **永続化の単位が明確**: 集約ごとにリポジトリを作れる（→ 次トピック）

## 3. 具体例

### 例①: Eコマース（注文 + 注文明細の古典例）

**悪い例: 集約内の要素を外部から直接触る**

```typescript
class Order {
  public lines: OrderLine[] = [];  // 外部から直接 push できてしまう
  public totalPrice: Money = Money.zero();
}

class OrderLine {
  public quantity: number = 1;  // 外部から書き換えできてしまう
}

// 外部コード（UI 層 など）
order.lines.push(new OrderLine(...));  // Order の不変条件チェックを迂回
order.lines[0].quantity = 999;         // totalPrice と不整合になる
```
→ 「合計は明細の合計と一致する」という不変条件が守られない。

**良い例: ルート経由で操作**

```typescript
class Order {  // 集約ルート
  private lines: OrderLine[] = [];

  constructor(
    readonly id: OrderId,
    readonly customerId: CustomerId,
  ) {}

  addLine(productId: ProductId, quantity: number, unitPrice: Money): void {
    if (quantity <= 0) throw new Error("数量は1以上");
    this.lines.push(new OrderLine(productId, quantity, unitPrice));
  }

  totalPrice(): Money {
    return this.lines.reduce(
      (acc, line) => acc.plus(line.subtotal()),
      Money.zero(),
    );
  }
}

class OrderLine {  // 集約内部。private 相当で外部公開しない
  constructor(
    readonly productId: ProductId,
    readonly quantity: number,
    readonly unitPrice: Money,
  ) {}

  subtotal(): Money {
    return this.unitPrice.times(this.quantity);
  }
}
```

> ポイント: `OrderLine` の追加・変更は必ず `Order` メソッド経由。外部は `order.addLine(...)` だけを知り、明細の内部表現や整合性ルールは Order に閉じる。

### 例②: 集約をまたぐ参照は ID で

**悪い例: 他集約を直接オブジェクト参照**

```typescript
class Order {
  constructor(
    readonly id: OrderId,
    readonly customer: Customer,  // Customer 集約を丸ごと保持
  ) {}
}
```
→ Order を読み込むたびに Customer もロードされ、メモリ肥大・整合性責任の混線が起きる。

**良い例: ID 参照で集約の独立性を保つ**

```typescript
class Order {
  constructor(
    readonly id: OrderId,
    readonly customerId: CustomerId,  // 他集約は ID のみ保持
  ) {}
}
```
→ 各集約は独立にロード・保存でき、境界がはっきりする。

## 4. よくある誤解・アンチパターン

### 誤解①: 「集約は大きいほど便利」
**誤り**: 「関連する Entity を全部 1 集約にすれば整合性が保てる」
**正しい考え方**: 大きすぎる集約は並行処理で競合しやすく、パフォーマンスも悪化する。**守るべき不変条件の最小単位** で集約を切るのが原則。

### 誤解②: 「集約内部の Entity にもリポジトリを作る」
**誤り**: 「`OrderLine` も DB テーブルがあるのでリポジトリを作る」
**正しい考え方**: リポジトリは **集約ルートに対してのみ** 作る。内部 Entity はルート経由で取得・変更する。

### 誤解③: 「集約をまたいでも即時一貫性を保てる」
**誤り**: 「注文と在庫は 1 トランザクションで整合させる」
**正しい考え方**: 集約をまたぐ整合性は **結果整合性（eventually consistent）** で設計するのが DDD の通例。ドメインイベント経由で非同期に整合させる（→ B 層「ドメインイベント」）。

### アンチパターン: God Aggregate（巨大集約）
全 Entity を 1 集約に詰め込み、毎回全体をロード・保存する。ロード/保存コスト爆発、ロック競合で並行性ゼロ、変更の影響範囲が不透明 → 典型的な失敗パターン。

## 5. 実装上の判断軸

### 集約として束ねるべきサイン
- 複数要素で同時に守るべき不変条件がある（「明細の合計 = 注文合計」など）
- 1 つのユースケースで必ず一緒に変更される
- ライフサイクルを共にする（削除時に連鎖的に消える）
- 業務上「一枚の紙（伝票）」として扱われる

### 別集約に分けるべきサイン
- 変更頻度やライフサイクルが異なる
- 別々に取得・更新したい場面が多い
- 結果整合性で十分な関係
- 既に独立の業務概念として確立されている（顧客と注文など）

### 集約の大きさを決めるチェックリスト
- [ ] このまとまり全体で守るべき不変条件を挙げられるか？（ないなら分けて良い）
- [ ] 1 つのユースケースでこれらを必ず同時変更するか？
- [ ] 別トランザクションに分けた場合、業務的に許容できない矛盾が起きるか？
- [ ] 他集約への参照は ID 1 本で済むか？（オブジェクト参照になっていたら境界が曖昧）

## 6. 関連する概念

- **[エンティティと値オブジェクト](エンティティと値オブジェクト.md)**: 集約は Entity と VO の集合として構成される。
- **リポジトリ**: 集約は集約単位で永続化する。リポジトリは集約ルートに対応する。
- **[境界づけられたコンテキスト](../../01_戦略的設計/説明/境界づけられたコンテキスト.md)**: 集約はコンテキスト内部の構造。境界を越えると集約も変わる。
- **ドメインイベント**（B層）: 集約をまたぐ結果整合性の担い手。
````

````markdown
# リポジトリ

## 1. 定義（〜とは）

**リポジトリ（Repository）** とは、**集約の永続化を抽象化するパターン**。集約のコレクションのように振る舞い、取得・保存・削除の操作を提供する。インターフェースはドメイン層に、具体実装はインフラ層に置く。

原典: Evans, DDD chapter 6。戦術的パターンの中で、ドメイン層とインフラ層の境界を定義する要。

> 一言メタファー: **図書館のカタログ**。本の保管場所（棚、倉庫、電子化）を利用者に意識させず、「この本を借りたい／返したい」だけで完結する。

## 2. なぜ必要か / 解決する問題

### リポジトリがない場合に起きる問題
- **ドメインが DB 実装に密結合**: ビジネスロジックの中で ORM 呼び出しや SQL が直接現れ、ドメイン層がインフラに依存する
- **テストが書きにくい**: ドメインロジックをテストするために DB 接続が必要になる
- **DB 変更時の影響範囲が読めない**: 永続化方式を変えるたびに、ドメイン全域を触ることになる
- **集約単位での保存が担保されない**: テーブル単位 CRUD で書くと、集約の整合性をまたいだ部分保存が起きる

### リポジトリが解決すること
- **ドメイン層は "集約を取得/保存する" という抽象にだけ依存**: 具体的な DB 実装を知らない
- **インフラ層で差し替え可能**: テスト用インメモリ、本番用 Postgres などを交換できる
- **集約単位の永続化が強制される**: 1 リポジトリ = 1 集約 の原則で、部分更新を防ぐ
- **ユビキタス言語を保てる**: `findPendingOrders()` のように業務語彙でメソッドを並べられる

## 3. 具体例

### 例①: Eコマース（OrderRepository）

**悪い例: ドメインが ORM を直接呼ぶ**

```typescript
class OrderService {
  async confirmOrder(orderId: string): Promise<void> {
    const row = await db.query("SELECT * FROM orders WHERE id = $1", [orderId]);
    // ...ドメインロジック...
    await db.query("UPDATE orders SET status = 'confirmed' WHERE id = $1", [orderId]);
  }
}
```
→ ドメインが SQL と結びつき、DB 変更やテストのたびに巻き添え。

**良い例: ドメイン層にインターフェース、インフラ層に実装**

```typescript
// ドメイン層: インターフェースのみ定義
interface OrderRepository {
  findById(id: OrderId): Promise<Order | null>;
  save(order: Order): Promise<void>;
  findPendingOrdersOf(customerId: CustomerId): Promise<Order[]>;
}

// インフラ層: 具体実装
class PostgresOrderRepository implements OrderRepository {
  constructor(private readonly db: Database) {}

  async findById(id: OrderId): Promise<Order | null> {
    const orderRow = await this.db.queryOne(
      "SELECT * FROM orders WHERE id = $1",
      [id.value],
    );
    if (!orderRow) return null;

    const lineRows = await this.db.query(
      "SELECT * FROM order_lines WHERE order_id = $1",
      [id.value],
    );
    return Order.reconstruct(orderRow, lineRows);  // 集約を復元
  }

  async save(order: Order): Promise<void> {
    // トランザクション内で集約全体（ヘッダ + 明細）を保存
  }
}
```

> ポイント: ドメイン層のコードは `OrderRepository` インターフェースだけを知り、`PostgresOrderRepository` の存在は知らない（依存関係逆転）。

### 例②: テスト用のインメモリ実装

```typescript
class InMemoryOrderRepository implements OrderRepository {
  private store = new Map<string, Order>();

  async findById(id: OrderId): Promise<Order | null> {
    return this.store.get(id.value) ?? null;
  }

  async save(order: Order): Promise<void> {
    this.store.set(order.id.value, order);
  }

  async findPendingOrdersOf(customerId: CustomerId): Promise<Order[]> {
    return [...this.store.values()].filter(
      (o) => o.customerId.equals(customerId) && o.isPending(),
    );
  }
}
```

> ポイント: ドメインロジックのテストでは `InMemoryOrderRepository` を注入できる。DB 起動なしで高速にテストが回る。

## 4. よくある誤解・アンチパターン

### 誤解①: 「リポジトリ = DAO」
**誤り**: 「リポジトリはテーブルごとの CRUD クラスだ」
**正しい考え方**: DAO はテーブル単位、リポジトリは **集約単位**。1 リポジトリが複数テーブルをまたいで 1 集約を構成することが珍しくない（Order と OrderLine のように）。

**正しい考え方**: リポジトリは **集約ルートに対してのみ** 作る。内部 Entity はルート経由で取得する。

### 誤解③: 「findByXxx を無制限に増やしていい」

**正しい考え方**: 検索条件が爆発するときは、**仕様パターン（Specification）** や **CQRS の読み取り側（クエリサービス）** を検討する（→ B 層・C 層）。リポジトリは本来「集約の取得・保存」が主務。

### アンチパターン: ジェネリックリポジトリの濫用
`Repository<T>` のような汎用基底で全集約を CRUD する構造。集約ごとの固有操作（`findPendingOrdersAwaitingPayment` など）が表現しづらくなり、結局別クラスに漏れ出す。

## 5. 実装上の判断軸

### リポジトリを作るべきサイン

- 読み取り負荷が書き込みと大きく違う（CQRS 検討）
````

````markdown
# ドメインサービス

## 1. 定義（〜とは）

**ドメインサービス（Domain Service）** とは、あるドメイン操作が特定の Entity や VO に自然に属さない場合に置く、**ステートレスな操作** を表現するパターン。ドメインの語彙で命名され、ドメイン層に属する。

原典: Evans, DDD chapter 5。Entity / VO に収まらないロジックの受け皿として位置づけられる。

> 一言メタファー: **窓口業務**。特定の口座オブジェクトの中に「送金」を書くのは不自然（どちらの口座に書くか？）。かといって UI 層に書くと送金ルールが散る。窓口は明確な責務を持つが、どの口座にも属さない。

## 2. なぜ必要か / 解決する問題

### ドメインサービスがない場合に起きる問題
- **複数集約にまたがる操作の置き場に困る**: 送金・交換・在庫割り当てを Entity に入れると、どちらかに不自然に偏る
- **アプリケーション層へ漏出**: 困ったロジックをアプリ層（ユースケース調整層）に置いてしまい、ドメインルールが散る
- **貧血モデルの逆側の失敗**: すべてを Entity に押し込もうとして、本質的に「2 者の関係性」で成立するロジックを Entity の中で強引に表現してしまう

### ドメインサービスが解決すること
- **ドメイン語彙のまま操作を置ける場所ができる**: 「送金（Transfer）」「割り当て（Allocation）」「照合（Reconciliation）」などが自然に表現できる
- **ドメイン層に留まる**: アプリ層には逃がさず、ユビキタス言語を保つ
- **ステートレスなので再利用・並行利用が安全**

## 3. 具体例

### 例①: Eコマース（注文割り当てサービス）

注文確定時、在庫集約から商品を確保して注文集約に割り当てる処理。**注文と在庫という 2 つの集約にまたがる**。

**悪い例: Order に押し込む**

```typescript
class Order {
  fulfill(inventory: Inventory): void {
    // Order が Inventory を直接触り始める → Order が肥大化し、
    // 責務が「注文の管理」から「在庫の管理」へ越境する
  }
}
```

**良い例: ドメインサービスとして抽出**

```typescript
class OrderAllocationService {
  allocate(order: Order, inventory: Inventory): AllocationResult {
    const shortages: ProductId[] = [];
    for (const line of order.lines()) {
      if (!inventory.hasAvailable(line.productId, line.quantity)) {
        shortages.push(line.productId);
      }
    }
    if (shortages.length > 0) {
      return AllocationResult.shortage(shortages);
    }
    order.lines().forEach((line) => inventory.reserve(line.productId, line.quantity));
    order.markAllocated();
    return AllocationResult.success();
  }
}
```

> ポイント: 「割り当て（Allocation）」という業務概念が明示的に存在し、注文・在庫どちらにも偏らない。将来、優先顧客向けの割当ポリシー等が入っても、このサービス内で拡張できる。

### 例②: 銀行の送金サービス（2 口座間の古典例）

```typescript
class TransferService {
  execute(source: Account, destination: Account, amount: Money): void {
    if (!source.currency().equals(destination.currency())) {
      throw new Error("currency mismatch（ForeignTransferService を使うこと）");
    }
    source.withdraw(amount);
    destination.deposit(amount);
  }
}
```

→ 「送金」は 2 口座の間に成立する概念で、どちらか一方の `Account` に置くと責務が歪む。為替や手数料が絡むと独立クラスとして育つ素地になる。

## 4. よくある誤解・アンチパターン

### 誤解①: 「`〇〇Service` はすべてドメインサービス」
**誤り**: 「名前が `Service` だからドメイン層」
**正しい考え方**: 名前が同じでも、**アプリケーションサービス（ユースケース調整）** と **ドメインサービス（ドメインルール）** は別物。ドメインサービスはドメイン語彙に限り、トランザクション制御や外部連携はアプリ層。

### 誤解②: 「ロジックが長いから切り出す」
**誤り**: 「Entity のメソッドが長くなったのでサービスに移す」
**正しい考え方**: まず Entity 内にリファクタで収まらないか検討。安易にサービスへ逃がすと、Entity がデータ容器化して **貧血ドメインモデル** になる。

### 誤解③: 「状態を持つドメインサービス」
**誤り**: 「キャッシュや進行状態を保持するサービスを作る」
**正しい考え方**: ドメインサービスは原則 **ステートレス**。状態が必要なら、それは Entity か VO ではないか、と問い直す。

### アンチパターン: Everything-is-a-Service
Entity にあるべき振る舞いをすべて `XxxService` に移した結果、Entity がデータ保持だけのクラスになる。DDD の目的（ビジネスルールをモデルに集約する）から逆行する。

## 5. 実装上の判断軸

### ドメインサービスにすべきサイン
- 2 つ以上の集約にまたがる操作（送金・割り当て・照合）
- どの Entity にも自然に属さない（主語が決められない）
- ドメイン語彙で名前が付く（「割り当てる」「送金する」「照合する」など）
- 本質的にステートレスな計算・判定

### ドメインサービスにすべきでないサイン（= 他に置く）
- 1 つの Entity/集約内で完結する → Entity メソッド
- トランザクション境界・リトライ・ログなど調整ロジック → アプリケーションサービス
- 外部 API 呼び出し・DB アクセス → インフラサービス
- 状態を保持する必要がある → Entity に格上げを検討

### 判断チェックリスト
- [ ] この操作は特定の Entity/VO に自然に属するか？ → Yes ならそちらへ
- [ ] 複数集約にまたがるか？ → Yes ならドメインサービス候補
- [ ] ドメイン語彙で命名できるか？ → No ならアプリ層の関心かも
- [ ] ステートレスに書けるか？ → No なら Entity 化を検討
````

````markdown
# レイヤードアーキテクチャ

## 1. 定義（〜とは）

**レイヤードアーキテクチャ（Layered Architecture）** とは、システムを責務別に層に分け、**依存方向を一方向に保つ** 構造。DDD の典型的な 4 層構成は以下:

| 層 | 責務 |
|----|------|
| **プレゼンテーション層** | ユーザー／外部 I/F との入出力（HTTP, UI） |
| **アプリケーション層** | ユースケース調整・トランザクション境界 |
| **ドメイン層** | ビジネスルール（Entity / VO / 集約 / ドメインサービス / リポジトリ I/F） |
| **インフラストラクチャ層** | DB, 外部 API, ファイル, ログなどの具体実装 |

原典: Evans, DDD chapter 4。依存方向の制御と、ドメイン層の独立性の確保が肝。

> 一言メタファー: **ビルのフロア分け**。受付（UI）→ 業務フロア（アプリ）→ 研究所（ドメイン）→ 電気・配管（インフラ）。各フロアの役割が混在しない。

## 2. なぜ必要か / 解決する問題

### 層分けがない場合に起きる問題
- **UI 変更でドメインが壊れる**: 関心が混ざり、変更影響が予測不能
- **ドメインがテスト不能になる**: Entity が HTTP や DB を直接触ると、単体テストに外部依存が必要
- **技術的関心がビジネスルールに紛れ込む**: SQL, HTTP, ORM の都合で Entity の設計が歪む
- **差し替えができない**: DB 変更や UI 差し替えのたびにドメインまで触る

### レイヤードアーキテクチャが解決すること
- **変更の影響範囲が層内に閉じやすい**
- **ドメイン層がインフラから独立** → 単体テストが容易
- **層の役割が名前で明確** → 新規メンバーの理解が早い
- **依存関係逆転（Dependency Inversion）** によりインフラはドメインに従属する

## 3. 具体例

### 例①: Eコマース 4 層分解

```
┌────────────────────────────────────┐
│ プレゼンテーション層               │
│ - HTTP ハンドラ / UI               │
│ - Next.js API Route, React         │
└────────────┬───────────────────────┘
             │ DTO / Request
┌────────────▼───────────────────────┐
│ アプリケーション層                 │
│ - ユースケース調整                 │
│ - PlaceOrderUseCase                │
└────────────┬───────────────────────┘
             │ Entity / 集約 / ドメインサービス
┌────────────▼───────────────────────┐
│ ドメイン層                         │
│ - Order / Customer / Money         │
│ - OrderRepository (interface)      │
│ - OrderAllocationService           │
└────────────▲───────────────────────┘
             │ interface を実装
┌────────────┴───────────────────────┐
│ インフラストラクチャ層             │
│ - PostgresOrderRepository          │
│ - Stripe 決済クライアント          │
└────────────────────────────────────┘
```

矢印方向: **上の層は下の層に依存してよい。逆は不可**。ドメイン層とインフラ層の関係は **依存関係逆転** によりインフラ→ドメイン向きになる。

```typescript
// ドメイン層
interface OrderRepository {
  save(order: Order): Promise<void>;
}

class Order { /* ... ビジネスルール ... */ }

// インフラ層（ドメイン層に依存）
class PostgresOrderRepository implements OrderRepository {
  async save(order: Order): Promise<void> { /* ... SQL ... */ }
}

// アプリケーション層（両方を組み立てる）
class PlaceOrderUseCase {
  constructor(private readonly orders: OrderRepository) {}  // interface に依存

  async execute(input: PlaceOrderInput): Promise<void> {
    const order = Order.place(input);
    await this.orders.save(order);
  }
}
```

> ポイント: `OrderRepository` の **定義** はドメイン層にあり、**実装** はインフラ層にある。ドメイン層はインフラ層を `import` しない（依存関係逆転）。

### 例②: 層違反の典型パターン

NG の例を並べて、即座に検出できる感覚を養う。

- **ドメイン層から `fetch` や SQL を呼ぶ**
  ```typescript
  class Order {
    async confirm() {
      await fetch("/api/payment", ...);  // ドメインが HTTP を知っている → NG
    }
  }
  ```
- **ドメイン層がインフラクラスを `import`**
  ```typescript
  import { PostgresClient } from "../infra/postgres";  // ドメイン → インフラの import → NG
  ```
- **アプリケーション層が HTTP レスポンスを直接組み立てる**（本来プレゼンテーション層の責務）
- **プレゼンテーション層に送料計算ロジック**（本来ドメイン層）

## 4. よくある誤解・アンチパターン

### 誤解①: 「層分け = ディレクトリ分け」
**誤り**: 「`domain/`, `application/`, `infra/` のディレクトリに分けたから層分け完了」
**正しい考え方**: 本質は **依存方向の制御**。ディレクトリを分けても双方向に import していたら意味がない。`tsconfig` の `paths` や ESLint ルールで依存方向を機械的に守るのが理想。

### 誤解②: 「ロガーくらいドメイン層に入れていい」
**誤り**: 「ログは便利だから Entity から呼ぶ」
**正しい考え方**: 厳密には NG。ログはインフラ関心。どうしても必要なら **ドメインイベント** を発火し、アプリ層またはインフラ層でログを取る。

### 誤解③: 「`〇〇Service` はすべてドメイン層」
**誤り**: 「Service という名前だからドメイン層」
**正しい考え方**: **ドメインサービス（ドメイン層）** と **アプリケーションサービス（アプリ層）** と **インフラサービス（インフラ層）** は名前が同じでも層が違う。責務で判断する。

### アンチパターン: 逆依存（ドメインがインフラを import）
ドメインが ORM・HTTP クライアントに依存すると、テスト不能・差し替え不能になる。依存関係逆転原則（DIP）を守り、インターフェースをドメインに置く。

### アンチパターン: 貧血な 4 層（Anemic Layers）
層は分けたが、ドメイン層に Entity の骨しかなく、全ロジックがアプリ層や UI に散る。名前だけの層分けで、ビジネスルールが依然として分散している状態。

## 5. 実装上の判断軸

### ドメイン層に置くべきサイン
- ビジネスルールそのものを担っている
- ユビキタス言語で命名できる
- 技術詳細（HTTP, DB, ファイル, ログ）に依存しない
- 企業のコア資産に近い知識

### アプリケーション層に置くべきサイン
- ユースケース単位の調整（複数のドメインサービス・リポジトリを組み合わせる）
- トランザクション境界・リトライ・認可チェック
- ドメインイベントのハンドラ登録

### インフラ層に置くべきサイン
- 外部システム（DB, API, キュー, ファイル）との通信
- リポジトリの具体実装
- フレームワーク依存のコード
````
