````markdown
# ドメインとサブドメイン

## 1. 定義（〜とは）

### ドメイン
**ドメイン（Domain）** とは、ソフトウェアが解決しようとしているビジネス領域・知識領域全体を指す。会社や組織が事業として扱っている領域そのものと言える。

### サブドメイン
**サブドメイン（Subdomain）** とは、大きなドメインを意味のある単位に分割した小領域。DDDではサブドメインを以下の3種類に分類する:

| 種類 | 説明 | 競争優位への寄与 |
|------|------|----------------|
| **コアドメイン**（Core Subdomain） | ビジネスの差別化要因となる最重要領域 | ⭐⭐⭐ 大 |
| **支援サブドメイン**（Supporting Subdomain） | コアを支えるが、それ自体は差別化要因ではない | ⭐⭐ 中 |
| **汎用サブドメイン**（Generic Subdomain） | どの会社でも共通の汎用機能 | ⭐ 小 |

> 一言メタファー: **コア=必殺技 / 支援=必殺技を成立させる準備 / 汎用=どこにでもある日用品**

## 2. なぜ必要か / 解決する問題

### サブドメイン分類がない場合に起きる問題
- **全機能を均等に扱ってしまう**: 差別化要因の機能と、どこにでもある機能（会計、認証など）を同じ熱量で作ってしまい、コアの品質が上がらない
- **「内製 vs 外部利用」の判断軸がない**: 本来SaaSや既製品で済ませるべき領域に時間を浪費する
- **チームのリソース配分が誤る**: 最も価値を生む領域に最も優秀な人材を配置できない

### サブドメイン分類が解決すること
- **力の入れどころが明確になる**: コアドメインに最も投資し、汎用サブドメインは外部サービスや既製品で済ませる判断ができる
- **DDDの戦術的パターン適用範囲を絞れる**: コアドメインにこそDDDの全戦術パターン（集約、ドメインサービス等）を適用し、汎用領域では軽量に作る

## 3. 具体例

### 例①: Eコマース
- **コアドメイン**: レコメンドエンジン（ユーザー体験の差別化要因）/ 注文管理（独自のビジネスルール）
- **支援サブドメイン**: 在庫管理 / 配送追跡
- **汎用サブドメイン**: 認証認可 / 決済処理 / 会計

> Eコマース企業にとって「決済」は重要だが、Stripe等の外部サービスを使うのが普通。一方「注文ルール」は自社の競争力に直結するため、丁寧にモデリングすべき。

### 例②: 物流会社
- **コアドメイン**: 配送ルート最適化アルゴリズム（コスト・速度の差別化要因）
- **支援サブドメイン**: 倉庫の在庫管理 / ドライバー管理
- **汎用サブドメイン**: 給与計算 / 経費精算

> 物流会社にとってルート最適化は他社との差別化点。給与計算は外部の人事SaaSで十分。

## 4. よくある誤解・アンチパターン

### 誤解①: 「機能の重要度＝コアドメイン」
**誤り**: 「ログイン機能はないと困るからコアドメインだ」
**正しい考え方**: ログインは「ないと困る」が、**他社との差別化要因にはならない** ので汎用サブドメイン。コアドメインは「他社との競争で勝つための領域」。

### 誤解②: 「サブドメイン＝マイクロサービスの単位」
**誤り**: 「サブドメインに分けたから、そのままマイクロサービスにしよう」
**正しい考え方**: サブドメインは **ビジネス上の関心の分割**。マイクロサービスは **技術的な実装の分割**。両者は近いが必ずしも一致しない（→ 詳細は「境界づけられたコンテキスト」へ）。

### 誤解③: 「汎用サブドメインは雑に作って良い」
**誤り**: 「会計部分は汎用だから雑なコードでOK」
**正しい考え方**: 「雑に作る」のではなく「**そもそも自社で作らず外部利用を検討する**」が正解。作らざるを得ないなら、シンプルに作って良い、という意味。

### アンチパターン: コアドメインへの投資不足
全領域を均等に作った結果、コアドメインに使える時間が減り、競争力のない凡庸なシステムになる。

## 5. 関連する概念

- **[境界づけられたコンテキスト](境界づけられたコンテキスト.md)**: サブドメイン（問題空間の分割）と、境界づけられたコンテキスト（解決空間の分割）は別物。次に読むのを推奨。
- **[ユビキタス言語](ユビキタス言語.md)**: 同じ言葉でも、サブドメインによって意味が変わる（例: Eコマースの「商品」は注文サブドメインと在庫サブドメインで違う側面を持つ）。
- **[DDDについて](../../DDDについて.md)**: DDDの全体概念に戻る。

### 次に読むべきもの
**[境界づけられたコンテキスト](境界づけられたコンテキスト.md)** へ進む。
````

````markdown
# 境界づけられたコンテキスト

## 1. 定義（〜とは）

**境界づけられたコンテキスト（Bounded Context）** とは、特定のドメインモデルが一貫した意味を持つ明確な境界を持った領域のこと。同じ言葉が別のコンテキストでは別の意味を持つことを認め、各コンテキスト内ではモデルと用語を厳密に統一する。

DDDの戦略的設計における **最重要概念**。Eric EvansのDDD原典でも中心的な位置づけ。

> 一言メタファー: **国境のような境界線**。同じ「自由」という言葉でも、国によって解釈が違う。境界の中ではルールと言葉が統一されている。

## 2. なぜ必要か / 解決する問題

### 境界づけられたコンテキストがない場合に起きる問題
- **全社統一モデル症候群**: 「商品」を全システム共通の1つのモデルにしようとして、注文・在庫・配送・マーケティングの全要件を1つのクラスに詰め込んだ結果、誰も理解できない巨大モデルになる
- **言葉の意味のすれ違い**: 営業部門の「顧客」と経理部門の「顧客」が違う意味なのに、同じ`Customer`クラスを共有してバグの温床になる
- **変更の影響範囲が爆発する**: 1つの概念を変更すると全システムに影響が及び、変更が怖くなる
- **チーム間の合意形成が永遠に終わらない**: 「商品とは何か」を全部門で合意しようとして、議論が決着しない

### 境界づけられたコンテキストが解決すること
- **「同じ言葉が別の意味を持つこと」を許容する**: 注文コンテキストの「商品」と在庫コンテキストの「商品」が違っていてOK
- **モデルの局所最適化が可能**: 各コンテキスト内では、その文脈に最適なモデルを作れる
- **チームの自律性向上**: コンテキスト境界がチーム境界とも対応しやすい
- **変更の影響範囲を境界内に閉じ込められる**

## 3. 具体例

### 例①: Eコマース

| コンテキスト | 「商品」のモデル |
|-------------|----------------|
| **注文管理** | 商品ID・名前・価格・購入可否 |
| **在庫管理** | 商品ID・SKU・倉庫別在庫数・入荷予定 |
| **配送管理** | 商品ID・サイズ・重量・配送区分（冷凍/常温） |
| **マーケティング** | 商品ID・カテゴリ・タグ・閲覧数・レコメンドスコア |

> 同じ「商品」でも、コンテキストごとに **必要な属性も振る舞いも全く違う**。すべてを1つの`Product`クラスに統合しようとすると破綻する。各コンテキストで独自の`Product`を持って良い（むしろ持つべき）。

### 例②: 病院システム

| コンテキスト | 「患者」のモデル |
|-------------|----------------|
| **外来診療** | 患者ID・主訴・診察履歴・次回予約 |
| **入院管理** | 患者ID・病棟・病室・入院日・担当医・食事制限 |
| **薬剤管理** | 患者ID・処方履歴・アレルギー・併用禁忌薬 |
| **会計** | 患者ID・保険情報・請求残高・支払い履歴 |

> 「患者」という同じ概念でも、外来では「来院する人」、入院では「ベッドを使う人」、薬剤では「投薬対象」と、扱う側面が大きく異なる。

## 4. よくある誤解・アンチパターン

### 誤解①: 「コンテキスト＝マイクロサービス」
**誤り**: 「マイクロサービスに分けたからDDDのコンテキスト分けができている」
**正しい考え方**: 境界づけられたコンテキストは **論理的なモデルの境界**。1つのモノリスの中に複数のコンテキストがあっても良いし、1つのコンテキストが複数のマイクロサービスで実装されてもおかしくない。ただし、コンテキスト境界とサービス境界を一致させるのは設計上の良い指針。

### 誤解②: 「サブドメイン＝コンテキスト」
**誤り**: 「サブドメインを分けたら、それがそのままコンテキストになる」
**正しい考え方**:
- **サブドメイン** = **問題空間（problem space）** の分割（ビジネス領域そのもの）
- **コンテキスト** = **解決空間（solution space）** の分割（ソフトウェアモデルの境界）

理想的には1サブドメイン=1コンテキストだが、組織や歴史的経緯により、1サブドメインが複数コンテキストにまたがることもある。

### 誤解③: 「全社統一マスタを作るべき」
**誤り**: 「商品マスタは全社で1つにすべきだ。その方がデータが綺麗になる」
**正しい考え方**: データ的に統一しても、各コンテキストで使う側面が違うため、結局コンテキストごとにビューやサブセットが必要になる。最初からコンテキストごとにモデルを分けた方が健全。

### アンチパターン: 巨大共有モデル（God Model）
全コンテキストで共有する巨大な`Product`/`Customer`/`Order`クラスを作り、すべての要件を詰め込む → 数百のフィールドを持ち、誰も全体を理解できないクラスになる。
````

````markdown
# ユビキタス言語

## 1. 定義（〜とは）

**ユビキタス言語（Ubiquitous Language）** とは、特定の境界づけられたコンテキスト内で、ドメインエキスパート・開発者・ステークホルダー全員が共通して使う統一された言語。会話、ドキュメント、コードのすべてで同じ用語を同じ意味で使う。

「ユビキタス（Ubiquitous）」=「遍在する・どこにでもある」の意。**コンテキスト内のあらゆる場所で同じ言葉が使われている状態** を指す。

> 一言メタファー: **チームの方言**。コンテキストを越えると通じない方言だが、コンテキスト内では誰もが流暢に話せる。

## 2. なぜ必要か / 解決する問題

### ユビキタス言語がない場合に起きる問題
- **翻訳コストが発生する**: ビジネス側が「契約」と呼ぶものを、開発者が「Subscription」と訳してコードに書く → 会議のたびに翻訳が必要
- **翻訳ミスが起きる**: 微妙にニュアンスが違う英訳を当てた結果、本来の業務と異なる挙動になる
- **コードからビジネスが読めない**: コードが技術用語ばかりで、ドメインエキスパートがレビューできない
- **ドキュメントとコードの言葉がズレる**: 設計書では「予約」、コードでは`Booking`、API名は`reservations` → どれが正しいか不明

### ユビキタス言語が解決すること
- **会話・コード・ドキュメントが1つの言語で書かれる**: 翻訳不要
- **ドメインエキスパートがコードを読める**: コードレビューや仕様確認に参加できる
- **要件と実装の乖離が起きにくい**: 言葉が同じだから、解釈のズレが起きにくい
- **ナレッジが言語に蓄積する**: 「この概念は何と呼ぶか」を決めること自体がドメイン理解を深める

## 3. 具体例

### 例①: Eコマース

| ビジネスでの呼び方 | コードでの表現 | DB列名 | 評価 |
|-------------------|--------------|--------|------|
| 注文 | `Order` | `orders` | ✅ 一致 |
| カート | `Cart` | `carts` | ✅ 一致 |
| 注文確定 | `Order.confirm()` | - | ✅ 一致 |

**良い例**: ビジネス会話「カートから注文確定するときに在庫チェックする」→ コード `cart.confirmOrder()` (内部で在庫チェック) → 言葉が一致。

**悪い例**: ビジネス「注文確定」だが、コードでは`Transaction.commit()` という汎用名 → 何の取引かわからない。

### 例②: 保険業界

保険業界は業界用語が独特で、ユビキタス言語が特に重要なドメイン。

| 用語 | 意味 | 注意点 |
|------|------|--------|
| **契約者** | 保険料を支払う人 | 「顧客」とは限らない |
| **被保険者** | 保険の対象となる人 | 契約者と同じこともあれば違うこともある |
| **受取人** | 保険金を受け取る人 | 上記2者とも違うことがある |
| **保険金** | 事故等で支払われる金 | 「保険料」（払うお金）と混同しがち |
| **保険料** | 契約者が払うお金 | 「保険金」とは逆向きのお金 |

> ここで `Customer` という汎用名を使うと、契約者・被保険者・受取人のどれを指すか不明になる。コードでも `Policyholder`（契約者）、`Insured`（被保険者）、`Beneficiary`（受取人）と業界用語そのままで命名するのがユビキタス言語の実践。

## 4. よくある誤解・アンチパターン

### 誤解①: 「全社で1つの統一言語を作るべき」
**誤り**: 「ユビキタス言語は全社で統一するもの」
**正しい考え方**: ユビキタス言語は **境界づけられたコンテキストごと** に成立する。コンテキストが違えば、同じ言葉が違う意味で良い（例: Eコマースで「商品」が注文管理と在庫管理で違うモデル）。

### 誤解②: 「英語に訳さなければいけない」
**誤り**: 「コードは英語だから、業界用語も英訳しなければ」
**正しい考え方**: 業界用語に良い英訳がない場合、**ローマ字や原語をそのまま使う方が安全**。例: 日本の不動産業界の「重要事項説明」を `ImportantMatterExplanation` と訳すより、`JuyojikoSetsumei` のようにする方が誤解が少ないこともある。

### 誤解③: 「言葉さえ統一すればOK」
**誤り**: 「クラス名を統一すれば、ユビキタス言語の実践完了」
**正しい考え方**: ユビキタス言語は **言葉だけでなく概念の意味も統一** する必要がある。「注文」というクラス名でも、人によって「カート確定後の状態」と思っているか「決済完了後の状態」と思っているかが違うと、結局すれ違いが起きる。

### アンチパターン: 開発者だけで決めた言葉
ドメインエキスパート不在で開発者が決めた命名は、ビジネス側の言葉とズレる → 結局会議で翻訳が必要に。**命名はドメインエキスパートと一緒に決める**のが原則。
````

````markdown
# ドメインモデル

## 1. 定義（〜とは）

**ドメインモデル（Domain Model）** とは、特定の境界づけられたコンテキストにおけるドメイン（業務領域）の概念・ルール・振る舞いを構造化した抽象表現。DDDでは **ドメインモデルがそのままコードに反映される** ことを重視する。

ドメインモデルは単なるデータ構造（DB設計）ではなく、**ビジネスルールや振る舞いを含んだ振る舞い付きモデル**である点が重要。

> 一言メタファー: **業務の地図**。地図は実物そのものではないが、業務を理解し議論するための共通の縮図となる。

## 2. なぜ必要か / 解決する問題

### ドメインモデルがない場合に起きる問題
- **データ中心設計の罠**: テーブル設計から始めて、ビジネスロジックがアプリケーション層やUI層に散らばる
- **貧血ドメインモデル（Anemic Domain Model）**: クラスにデータだけあって振る舞いがない → ビジネスルールがどこにあるか分からない
- **同じビジネスルールが各所に散在**: 「注文金額10,000円以上で送料無料」というルールが、フロントエンド・バックエンドAPI・バッチ処理・帳票生成に重複して書かれる → 1つを変更し忘れてバグになる
- **コードからビジネスが読めない**: メソッド名が`updateData()`のような汎用名で、何のビジネス操作か分からない

### ドメインモデルが解決すること
- **ビジネスルールがモデル内に集約される**: 同じルールが1箇所にある状態が保たれる
- **コード自体がドキュメントになる**: `order.confirm()`、`account.withdraw(amount)` のように、コードを読めばビジネス操作が分かる
- **ドメインエキスパートとの議論の土台になる**: モデル図やクラス名がそのままビジネス概念を反映するので、議論しやすい
- **変更がモデル中心で進む**: ビジネスルール変更 → モデル変更 → コード変更、という一直線の流れになる

## 3. 具体例

### 例①: Eコマース

**ビジネスルール**: 「注文金額10,000円以上で送料無料、それ以下は500円」「会員ランクGoldは常に送料無料」

#### 悪い例: 振る舞いがUI層に散らばる
```
// UI層で計算
if (order.totalPrice >= 10000 || user.rank === "Gold") {
  shipping = 0;
} else {
  shipping = 500;
}
```
→ 同じ計算がバッチ・帳票・APIにも書かれて重複する。

#### 良い例: ドメインモデル内に閉じる
```
class Order {
  calculateShippingFee(customer: Customer): Money {
    if (customer.isGoldMember()) return Money.zero();
    if (this.totalPrice() >= 10000) return Money.zero();
    return Money.yen(500);
  }
}
```
→ 送料計算ルールはOrderモデルだけが知っている。変更時はここだけ修正すればよい。

### 例②: 銀行口座（モデリングの古典例）

**ビジネスルール**:
- 残高がマイナスになる出金は不可（普通預金の場合）
- 当座預金は1日100万円まで透支（マイナス）可能
- 入出金は記録され、後から履歴参照できる

#### ドメインモデルとして表現
```
class Account {
  private balance: Money;
  private history: Transaction[];

  withdraw(amount: Money): void {
    if (this.balance.minus(amount).isLessThan(this.overdraftLimit())) {
      throw new InsufficientFundsError();
    }
    this.balance = this.balance.minus(amount);
    this.history.push(Transaction.withdrawal(amount));
  }

  deposit(amount: Money): void {
    this.balance = this.balance.plus(amount);
    this.history.push(Transaction.deposit(amount));
  }

  protected overdraftLimit(): Money {
    return Money.zero();  // 普通預金は0、当座預金はoverride
  }
}

class CheckingAccount extends Account {
  protected overdraftLimit(): Money {
    return Money.yen(-1_000_000);  // 当座預金は-100万円まで
  }
}
```

> ポイント: **「出金できるか」のルールが Account モデル内に閉じている**。UI層やAPI層では`account.withdraw(amount)`を呼ぶだけで、ルールはモデルが守る。

## 4. よくある誤解・アンチパターン

### 誤解①: 「ドメインモデル＝ER図/DBスキーマ」
**誤り**: 「テーブル設計＝ドメインモデル」
**正しい考え方**: DBスキーマは **データの永続化形式**。ドメインモデルは **ビジネス概念と振る舞いの構造**。両者は近いことも多いが、別物。例えば、ドメインモデルでは「不変オブジェクト」で表現する概念を、DBでは複数テーブルに分割保存することもある。

### 誤解②: 「貧血ドメインモデルでもDDDと言える」
**誤り**: 「`Order`クラスを作って、ビジネスロジックは`OrderService`に書いた。これでDDDだ」
**正しい考え方**: データだけ持って振る舞いがないクラスを **貧血ドメインモデル（Anemic Domain Model）** と呼び、DDDのアンチパターンとされる。ビジネスロジックがサービス層に散ってしまうため、DDDの目的（モデルにビジネスを集約）が達成できない。

### 誤解③: 「全コンテキストで共通のドメインモデルを作るべき」
**誤り**: 「全社共通の`Customer`モデルを作って、各部門が共有しよう」
**正しい考え方**: ドメインモデルは **境界づけられたコンテキストごと** に存在する。同じ顧客でも、営業コンテキストと配送コンテキストで違うモデルを持って良い（むしろ持つべき）。

### アンチパターン: トランザクションスクリプト
全ロジックを「処理手順の関数」として書き、ドメインモデル不在になるパターン。小規模アプリには有効だが、複雑なビジネスロジックには不向き。

## 5. 関連する概念

- **[ユビキタス言語](ユビキタス言語.md)**: ドメインモデルのクラス名・メソッド名はユビキタス言語そのもの。
- **[境界づけられたコンテキスト](境界づけられたコンテキスト.md)**: ドメインモデルはコンテキストごとに存在する。
- **エンティティ / 値オブジェクト / 集約**（A層）: ドメインモデルを実装する戦術的パターン。本資料ではA層で扱う。
- **[DDDについて](../../DDDについて.md)**: 「モデルと実装の一致」の節を再読すると理解が深まる。
````
