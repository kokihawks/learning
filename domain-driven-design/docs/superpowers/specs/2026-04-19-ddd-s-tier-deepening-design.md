# DDD資料 S層深掘り 設計書

## 目的

既存の `DDDについて.md`（DDDの概念紹介レベル）を起点に、DDD理解に必要な知識のうち**戦略的設計（Strategic Design）= 優先度S層**を深掘りした学習資料を整備する。これにより、初学者が「DDDとは何か」だけでなく「DDDをどう設計に落とすか」の入口に立てるようにする。

## スコープ

### 対象（やること）
- S層4トピックを1ファイルずつ作成
  1. ドメインとサブドメイン
  2. 境界づけられたコンテキスト
  3. ユビキタス言語
  4. ドメインモデル
- 学習ロードマップとなる `知識の優先度マップ.md` を新規作成
- 既存 `DDDについて.md` の末尾に、マップへの導線を1節だけ追加

### 対象外（やらないこと）
- A層（戦術的設計：Entity/Value Object/Aggregate等）の作成
- B層・C層の作成
- 既存 `DDDについて.md` 本文の書き換え
- コードサンプル（実装パターンに踏み込むのはA層以降）

## ディレクトリ構成

```
domain-driven-design/
├ DDDについて.md                          ← 既存（DDDの概念紹介・入口）
├ 知識の優先度マップ.md                    ← 新規（学習ロードマップ・全体像）
│
└ 01_戦略的設計/
    └ 説明/
        ├ ドメインとサブドメイン.md         ← 新規 ①
        ├ 境界づけられたコンテキスト.md     ← 新規 ②
        ├ ユビキタス言語.md                ← 新規 ③
        └ ドメインモデル.md                ← 新規 ④
```

- AWS資料側の `技術領域/XX_カテゴリ/説明/トピック.md` 流儀に揃える
- `02_戦術的設計/` 等は本スコープでは作成しない（YAGNI）

## 各トピックファイルの構成テンプレート

S層4ファイルすべてを以下の **学習特化型テンプレート** で統一する。

```markdown
# [トピック名]

## 1. 定義（〜とは）
- 簡潔な定義（1-2文）
- DDD原典での位置づけ
- 一言メタファー（必要なら）

## 2. なぜ必要か / 解決する問題
- このトピックが存在しない場合に起きる問題
- どんな設計上の課題に対する答えなのか
- 関連する従来手法との違い

## 3. 具体例
- 具体例 ①: Eコマース（既存資料との一貫性）
- 具体例 ②: もう1ドメイン（業種を変えた例）
- 各例で「DDDのこの概念がどう現れるか」を示す

## 4. よくある誤解・アンチパターン
- 初学者がハマりがちな誤解
- 「〜と混同されやすいが実際は違う」
- 実装時の落とし穴

## 5. 関連する概念
- このトピックと密接に関わる他のDDD概念
- 他トピックファイルへのリンク
- 「次に読むべきもの」の誘導
```

### 例ドメインの選定

- **メイン例: Eコマース**（既存資料との一貫性を保つため全トピックで使用）
- **サブ例: トピックごとに使い分け**
  | トピック | サブ例 | 選定理由 |
  |---------|--------|---------|
  | ドメインとサブドメイン | 物流会社 | コア/支援/汎用が明確に分かれる典型例 |
  | 境界づけられたコンテキスト | 病院/医療システム | コンテキスト分割が直感的 |
  | ユビキタス言語 | 保険業界 | 業界用語が独特で誤解事例が豊富 |
  | ドメインモデル | 銀行口座 | モデル化の古典例 |

## `知識の優先度マップ.md` の構成

```markdown
# DDD知識の優先度マップ

## このドキュメントの目的
DDDは概念が多く、学習者が「何から手をつけるべきか」で迷いやすい。
本ドキュメントでは、DDD理解に必要な知識を4段階の優先度に分類し、
学習順序の指針を示す。

## 優先度の凡例
- 🔴 S: 最重要（これを知らずしてDDDは理解できない）
- 🟠 A: 重要（実装に踏み込むなら必須）
- 🟡 B: 応用（複雑な設計判断に必要）
- 🟢 C: 発展（DDDを使いこなす段階）

## 🔴 優先度 S（戦略的設計の根幹）
| # | トピック | 状態 | リンク |
|---|---------|------|--------|
| 1 | ドメインとサブドメイン | 🚧/⏳/✅ | [→](...) |
| 2 | 境界づけられたコンテキスト | ⏳ | - |
| 3 | ユビキタス言語 | ⏳ | - |
| 4 | ドメインモデル | ⏳ | - |

## 🟠 優先度 A（戦術的設計：実装パターン）
（5〜9のトピック表。リンクは未作成のためなし）

## 🟡 優先度 B（応用パターン）
（10〜14のトピック表）

## 🟢 優先度 C（発展トピック）
（15〜19のトピック表）

## 推奨学習順序
S層 → A層 → B層 → C層 を基本とする。
ただし(補足)...

## 関連ドキュメント
- [DDDについて](DDDについて.md) - DDDの概念紹介
```

### 進捗管理
- **状態カラム**で可視化: 🚧 作成中 / ⏳ 未着手 / ✅ 完成
- 各トピックファイルが完成するたびにマップを更新

## 作業順序

1. `知識の優先度マップ.md` を新規作成（S層のリンクは未作成状態で表に枠だけ用意）
2. `01_戦略的設計/説明/ドメインとサブドメイン.md` を作成
3. マップを更新（①を ✅、リンクを有効化）
4. `境界づけられたコンテキスト.md` を作成 → マップ更新
5. `ユビキタス言語.md` を作成 → マップ更新
6. `ドメインモデル.md` を作成 → マップ更新
7. `DDDについて.md` の末尾に「次に読むべきもの」節を追加してマップへリンク

## コミット粒度

- **1トピック1コミット**（例: `S層: ドメインとサブドメイン.md を作成`）
- マップ更新は同コミットに含める
- 最後の `DDDについて.md` への導線追加は別コミット

## 成功基準

- S層4ファイルすべてが学習特化型テンプレートに沿って作成されていること
- `知識の優先度マップ.md` から各トピックへ辿れること
- `DDDについて.md` を入口として、マップ → 各トピックという学習動線が成立していること
- 各トピックに「具体例」と「よくある誤解」が必ず含まれていること

## やらないこと（再掲・明示）

- コードサンプル
- A層以降のディレクトリ作成
- 既存 `DDDについて.md` の本文書き換え
- 図表・ダイアグラム（テキストのみで完結させる）
