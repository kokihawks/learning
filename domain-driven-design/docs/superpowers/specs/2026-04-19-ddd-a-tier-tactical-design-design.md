# DDD資料 A層（戦術的設計）深掘り 設計書

## 目的

S層（戦略的設計）4トピックが `01_戦略的設計/説明/` 配下に完成し、`知識の優先度マップ.md` から辿れる状態になった。本設計はその次段、**優先度 A 層＝戦術的設計（Tactical Design）** の5トピックを深掘りした学習資料を整備する。

S層で「DDDをどう分けるか」を理解した学習者を、A層では「ドメインモデルを**どう実装するか**」の段階に引き上げる。現場で実装判断を下せるようになることをゴールに置く。

## スコープ

### 対象（やること）
- A層5トピックを1ファイルずつ作成
  1. エンティティと値オブジェクト
  2. 集約と集約ルート
  3. リポジトリ
  4. ドメインサービス
  5. レイヤードアーキテクチャ
- 既存 `知識の優先度マップ.md` のA層テーブルにリンク列を追加し、各状態を ✅ に更新
- 既存 `01_戦略的設計/説明/ドメインモデル.md` 末尾「次に読むべきもの」にA層先頭への具体リンクを追加
- `DDDについて.md` の戦術的設計の節に、マップのA層セクションへの相対リンクを必要に応じて加筆

### 対象外（やらないこと）
- B層（コンテキストマップ、ドメインイベント、仕様、ヘキサゴナル等）の作成
- C層（CQRS、イベントソーシング、マイクロサービス等）の作成
- 実行可能なテスト環境の整備（package.json / tsconfig / vitest 等）
- 図表・ダイアグラム（テキストとコードブロックのみで完結させる）
- 既存 S 層ファイル本文の書き換え（リンク追加のみ）

## 各トピックファイルの構成テンプレート

S層の5節テンプレートに **A層固有の第5節「実装上の判断軸」を追加** した **6節構成** で統一する。

```markdown
# [トピック名]

## 1. 定義（〜とは）
- 簡潔な定義（1-2文）
- DDD原典での位置づけ
- 一言メタファー

## 2. なぜ必要か / 解決する問題
- このパターンが存在しない場合に起きる問題
- どんな設計上の課題に対する答えなのか

## 3. 具体例
- 具体例 ①: Eコマース（S層から文脈を引き継ぐメイン例）
  - 悪い例 → 良い例 の対比を TypeScript スケッチで示す
- 具体例 ②: トピックごとに最適なドメイン
  - 当該パターンの本質が最もクリアに出る題材を選ぶ

## 4. よくある誤解・アンチパターン
- 初学者がハマりがちな誤解（3項目）
- 名前が似た別概念との混同
- 実装時の落とし穴（アンチパターン名付き）

## 5. 実装上の判断軸  ← A層固有
### このパターンを採用すべきサイン
### 採用を避けるべきサイン（別パターンが適切）
### 判断に迷ったときのチェックリスト

## 6. 関連する概念
- 密接に関わる他のDDD概念
- 他トピックファイルへのリンク
- 「次に読むべきもの」の誘導
```

### 第5節を新設する理由

S層は「思想・概念」が中心で、読んで腹落ちすれば達成。しかしA層は **現場で「どのパターンを選ぶか」の意思決定** が本質である。パターンの定義と例だけを並べても「知識はあるが使えない」状態になりやすい。採用／非採用のサインと判断チェックリストを各トピックに必ず置くことで、学習者が業務で迷ったときにその節だけ見返せば判断できる構造にする。

### コード例の方針

- **言語**: TypeScript（S層 `ドメインモデル.md` と統一）
- **深さ**: 読めば分かるスケッチ。テストや実行環境は付けない
- **個数**: 1トピックあたり 2〜3 個（悪い例・良い例・補足例）
- **共通語彙**: `Money`, `Order`, `Customer`, `Address` 等、S層で既出のクラス名を再利用してシリーズ感を出す
- **1スニペット目安**: 15〜25行。冗長な import やセッター定義は省略する

## 例ドメインの選定

- **メイン例: Eコマース**（S層から続投、全トピックで使用）
- **サブ例: トピックごとに使い分け**
  | # | トピック | サブ例 | 選定理由 |
  |---|---------|--------|---------|
  | 5 | エンティティと値オブジェクト | 住所（VO）/ 顧客（Entity） | 同一性 vs 属性等価性の対比が最もクリア |
  | 6 | 集約と集約ルート | 注文 + 注文明細 | 集約の古典例。トランザクション境界が直感的 |
  | 7 | リポジトリ | 注文リポジトリ | 集約単位での永続化がそのまま説明できる |
  | 8 | ドメインサービス | 送金（2口座間） | Entity にも VO にも置けないロジックの典型 |
  | 9 | レイヤードアーキテクチャ | Eコマース全体を 4 層に分解 | 既出概念を層別に整理する総集編的位置づけ |

## 5トピックの中心問い

執筆時の道標として、各トピックの「読み終わった学習者が答えられるべき問い」を事前に固定する。

| # | トピック | 中心問い |
|---|---------|---------|
| 5 | エンティティと値オブジェクト | 同一性は ID で追うべきか、属性で決まるか |
| 6 | 集約と集約ルート | トランザクション境界をどこに引くか、集約の大きさをどう決めるか |
| 7 | リポジトリ | 集約をどう永続化・再構築するか、DB 実装をどう隠すか |
| 8 | ドメインサービス | Entity にも VO にも置けないロジックの置き場 |
| 9 | レイヤードアーキテクチャ | ドメイン層は何に依存してよいか、依存方向をどう制御するか |

## 作業順序

**優先度マップ記載の順（5→6→7→8→9）** で進める。依存関係が自然に流れる順序:
単体（Entity/VO）→ 集合（Aggregate）→ 永続化（Repository）→ 横断ロジック（Domain Service）→ 全体構造（Layered）。

1. `02_戦術的設計/説明/` ディレクトリを作成
2. `エンティティと値オブジェクト.md` を作成 → マップ更新（⑤を ✅、リンク有効化）
3. `集約と集約ルート.md` を作成 → マップ更新（⑥）
4. `リポジトリ.md` を作成 → マップ更新（⑦）
5. `ドメインサービス.md` を作成 → マップ更新（⑧）
6. `レイヤードアーキテクチャ.md` を作成 → マップ更新（⑨）
7. `ドメインモデル.md` 末尾「次に読むべきもの」にA層先頭への具体リンクを追加
8. `DDDについて.md` の戦術的設計の節を確認し、必要ならマップのA層セクションへの相対リンクを加筆

## コミット粒度

S層と同じく **1トピック1コミット** とする（例: `DDD A層: エンティティと値オブジェクト.md を作成`）。マップ更新は同コミットに含める。最後の S 層ファイルと `DDDについて.md` への導線追加は別コミットで分離する。

## 成功基準

- A層5ファイルすべてが 6 節テンプレ（**特に第5節「実装上の判断軸」**）に沿って作成されていること
- `知識の優先度マップ.md` の A 層テーブルで 5 行すべてが ✅ になり、リンクが有効であること
- `DDDについて.md` → マップ → A層トピック5→9 → `ドメインモデル.md` と相対リンクで往復できること
- 5 ファイル中 3 ファイル以上でメイン例に Eコマース（`Order` / `Customer` 等）が登場していること（シリーズ感の担保）
- 各トピックが 100〜150 行目安に収まっていること（判断軸節ぶんで S 層より 30 行程度厚い想定）

## やらないこと（再掲・明示）

- B 層・C 層の作成
- 図表・ダイアグラム
- 実行可能なテスト／ビルド環境
- S 層既存ファイル本文の書き換え（末尾のリンク追加のみ許可）
- 「vs」表記の導入（ファイル名・見出しは「と」で統一）
