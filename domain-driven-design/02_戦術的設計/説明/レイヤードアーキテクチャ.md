# レイヤードアーキテクチャ

## 1. 定義（〜とは）

**レイヤードアーキテクチャ（Layered Architecture）** とは、システムを責務別に層に分け、**依存方向を一方向に保つ** 構造。DDD の典型的な 4 層構成は以下:

| 層 | 責務 |
|----|------|
| **プレゼンテーション層** | ユーザー／外部 I/F との入出力（HTTP, UI） |
| **アプリケーション層** | ユースケース調整・トランザクション境界 |
| **ドメイン層** | ビジネスルール（Entity / VO / 集約 / ドメインサービス / リポジトリ I/F） |
| **インフラストラクチャ層** | DB, 外部 API, ファイル, ログなどの具体実装 |

原典: Evans, DDD chapter 4。依存方向の制御と、ドメイン層の独立性の確保が肝。

> 一言メタファー: **ビルのフロア分け**。受付（UI）→ 業務フロア（アプリ）→ 研究所（ドメイン）→ 電気・配管（インフラ）。各フロアの役割が混在しない。

## 2. なぜ必要か / 解決する問題

### 層分けがない場合に起きる問題
- **UI 変更でドメインが壊れる**: 関心が混ざり、変更影響が予測不能
- **ドメインがテスト不能になる**: Entity が HTTP や DB を直接触ると、単体テストに外部依存が必要
- **技術的関心がビジネスルールに紛れ込む**: SQL, HTTP, ORM の都合で Entity の設計が歪む
- **差し替えができない**: DB 変更や UI 差し替えのたびにドメインまで触る

### レイヤードアーキテクチャが解決すること
- **変更の影響範囲が層内に閉じやすい**
- **ドメイン層がインフラから独立** → 単体テストが容易
- **層の役割が名前で明確** → 新規メンバーの理解が早い
- **依存関係逆転（Dependency Inversion）** によりインフラはドメインに従属する

## 3. 具体例

### 例①: Eコマース 4 層分解

```
┌────────────────────────────────────┐
│ プレゼンテーション層               │
│ - HTTP ハンドラ / UI               │
│ - Next.js API Route, React         │
└────────────┬───────────────────────┘
             │ DTO / Request
┌────────────▼───────────────────────┐
│ アプリケーション層                 │
│ - ユースケース調整                 │
│ - PlaceOrderUseCase                │
└────────────┬───────────────────────┘
             │ Entity / 集約 / ドメインサービス
┌────────────▼───────────────────────┐
│ ドメイン層                         │
│ - Order / Customer / Money         │
│ - OrderRepository (interface)      │
│ - OrderAllocationService           │
└────────────▲───────────────────────┘
             │ interface を実装
┌────────────┴───────────────────────┐
│ インフラストラクチャ層             │
│ - PostgresOrderRepository          │
│ - Stripe 決済クライアント          │
└────────────────────────────────────┘
```

矢印方向: **上の層は下の層に依存してよい。逆は不可**。ドメイン層とインフラ層の関係は **依存関係逆転** によりインフラ→ドメイン向きになる。

```typescript
// ドメイン層
interface OrderRepository {
  save(order: Order): Promise<void>;
}

class Order { /* ... ビジネスルール ... */ }

// インフラ層（ドメイン層に依存）
class PostgresOrderRepository implements OrderRepository {
  async save(order: Order): Promise<void> { /* ... SQL ... */ }
}

// アプリケーション層（両方を組み立てる）
class PlaceOrderUseCase {
  constructor(private readonly orders: OrderRepository) {}  // interface に依存

  async execute(input: PlaceOrderInput): Promise<void> {
    const order = Order.place(input);
    await this.orders.save(order);
  }
}
```

> ポイント: `OrderRepository` の **定義** はドメイン層にあり、**実装** はインフラ層にある。ドメイン層はインフラ層を `import` しない（依存関係逆転）。

### 例②: 層違反の典型パターン

NG の例を並べて、即座に検出できる感覚を養う。

- **ドメイン層から `fetch` や SQL を呼ぶ**
  ```typescript
  class Order {
    async confirm() {
      await fetch("/api/payment", ...);  // ドメインが HTTP を知っている → NG
    }
  }
  ```
- **ドメイン層がインフラクラスを `import`**
  ```typescript
  import { PostgresClient } from "../infra/postgres";  // ドメイン → インフラの import → NG
  ```
- **アプリケーション層が HTTP レスポンスを直接組み立てる**（本来プレゼンテーション層の責務）
- **プレゼンテーション層に送料計算ロジック**（本来ドメイン層）

## 4. よくある誤解・アンチパターン

### 誤解①: 「層分け = ディレクトリ分け」
**誤り**: 「`domain/`, `application/`, `infra/` のディレクトリに分けたから層分け完了」
**正しい考え方**: 本質は **依存方向の制御**。ディレクトリを分けても双方向に import していたら意味がない。`tsconfig` の `paths` や ESLint ルールで依存方向を機械的に守るのが理想。

### 誤解②: 「ロガーくらいドメイン層に入れていい」
**誤り**: 「ログは便利だから Entity から呼ぶ」
**正しい考え方**: 厳密には NG。ログはインフラ関心。どうしても必要なら **ドメインイベント** を発火し、アプリ層またはインフラ層でログを取る。

### 誤解③: 「`〇〇Service` はすべてドメイン層」
**誤り**: 「Service という名前だからドメイン層」
**正しい考え方**: **ドメインサービス（ドメイン層）** と **アプリケーションサービス（アプリ層）** と **インフラサービス（インフラ層）** は名前が同じでも層が違う。責務で判断する。

### アンチパターン: 逆依存（ドメインがインフラを import）
ドメインが ORM・HTTP クライアントに依存すると、テスト不能・差し替え不能になる。依存関係逆転原則（DIP）を守り、インターフェースをドメインに置く。

### アンチパターン: 貧血な 4 層（Anemic Layers）
層は分けたが、ドメイン層に Entity の骨しかなく、全ロジックがアプリ層や UI に散る。名前だけの層分けで、ビジネスルールが依然として分散している状態。

## 5. 実装上の判断軸

### ドメイン層に置くべきサイン
- ビジネスルールそのものを担っている
- ユビキタス言語で命名できる
- 技術詳細（HTTP, DB, ファイル, ログ）に依存しない
- 企業のコア資産に近い知識

### アプリケーション層に置くべきサイン
- ユースケース単位の調整（複数のドメインサービス・リポジトリを組み合わせる）
- トランザクション境界・リトライ・認可チェック
- ドメインイベントのハンドラ登録

### インフラ層に置くべきサイン
- 外部システム（DB, API, キュー, ファイル）との通信
- リポジトリの具体実装
- フレームワーク依存のコード

### 判断チェックリスト
- [ ] このクラスが `import` しているモジュールは、自分より下位の層のみか？（上位を import したら層違反）
- [ ] ドメイン層のコードは SQL / HTTP / ORM / ファイルパスを知らないか？
- [ ] ユースケース調整（トランザクション、ログ、リトライ）がドメイン層に漏れていないか？
- [ ] プレゼンテーション層にビジネスルール（送料計算、割引判定など）が紛れていないか？
