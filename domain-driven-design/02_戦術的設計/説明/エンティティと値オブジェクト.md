# エンティティと値オブジェクト

## 1. 定義（〜とは）

### エンティティ
**エンティティ（Entity）** とは、ライフサイクルを通じて **一意な同一性（identity）** を持つオブジェクト。属性が変わっても「同じ〇〇である」と追跡され続ける。識別は ID で行う。

### 値オブジェクト
**値オブジェクト（Value Object, VO）** とは、**属性の組そのもの** を表す不変オブジェクト。属性が全て等しければ交換可能とみなす。ID を持たず、同一性ではなく等価性で比較する。

DDD 原典では、両者は戦術的設計の最も基礎的なビルディングブロックとして位置づけられる（Evans, DDD chapter 5）。

> 一言メタファー: **エンティティ=人**（名前や住所が変わっても同じ人）、**値オブジェクト=お金**（1000円札と1000円札は交換可能）

## 2. なぜ必要か / 解決する問題

### 区別がない場合に起きる問題
- **全てをエンティティで扱う**: 住所や金額にまで ID を付け、同じ内容なのに別物扱いされる。無駄なライフサイクル管理が発生し、ドメインが複雑化する
- **全てをただのデータ構造で扱う**: 不変性が保証されず、副作用が読めない。型が `string` や `number` ばかりになり、通貨違いや住所の不整合が実行時まで検知できない（Primitive Obsession）
- **同一性の基準が曖昧**: 「同じ顧客か」を判定する基準がコードごとに違う（名前一致 / ID一致 / メール一致）

### 区別が解決すること
- **同一性 vs 等価性の意図がコードで宣言される**: 何を以て "同じ" とするかが明確
- **VO は不変でスレッドセーフ**: 副作用が減り、計算結果を合成しやすい
- **エンティティのライフサイクル管理が一点集中**: 追跡対象が明確になる

## 3. 具体例

### 例①: Eコマース（住所 vs 顧客）

**悪い例: すべてをエンティティで扱う**

```typescript
class Address {
  constructor(
    public id: string,
    public postalCode: string,
    public city: string,
    public street: string,
  ) {}
}
```
→ 同じ「東京都渋谷区〇〇 1-2-3」でも ID が違えば別物扱い。顧客Aと顧客Bが同じ住所を共有しているか検知できず、重複判定も効かない。

**良い例: 住所は VO、顧客は Entity**

```typescript
class Address {  // 値オブジェクト
  constructor(
    readonly postalCode: string,
    readonly city: string,
    readonly street: string,
  ) {}

  equals(other: Address): boolean {
    return this.postalCode === other.postalCode
      && this.city === other.city
      && this.street === other.street;
  }
}

class Customer {  // エンティティ
  constructor(
    readonly id: CustomerId,
    private name: string,
    private address: Address,
  ) {}

  moveTo(newAddress: Address): void {
    this.address = newAddress;  // Address は丸ごと差し替え（不変）
  }

  equals(other: Customer): boolean {
    return this.id.equals(other.id);  // 同一性は ID で判定
  }
}
```

> ポイント: **住所は属性の束であり、内容が等しければ "同じ住所" と扱える**。顧客は ID で追跡され、引っ越し（属性変更）しても同じ顧客として残る。

### 例②: 金額（Money）— 典型的な VO

```typescript
class Money {
  private constructor(
    readonly amount: number,
    readonly currency: string,
  ) {}

  static yen(amount: number): Money { return new Money(amount, "JPY"); }
  static zero(): Money { return new Money(0, "JPY"); }

  plus(other: Money): Money {
    if (this.currency !== other.currency) {
      throw new Error("currency mismatch");
    }
    return new Money(this.amount + other.amount, this.currency);
  }

  equals(other: Money): boolean {
    return this.amount === other.amount && this.currency === other.currency;
  }
}
```

> ポイント: `Money.yen(1000).equals(Money.yen(1000))` は `true`。計算操作は必ず新しい `Money` を返す（不変）。`number` 単独では表現できない「通貨の一致」をコンパイル時に強制できる。

## 4. よくある誤解・アンチパターン

### 誤解①: 「ID があればエンティティ、なければ VO」
**誤り**: 「このクラスには `id` フィールドがあるからエンティティだ」
**正しい考え方**: ID の有無ではなく、**同一性をどう扱うか** が本質。DB のサロゲートキーのために ID を付けただけのクラスは VO として扱うべき場合がある。

### 誤解②: 「金額や日付はプリミティブで十分」
**誤り**: 「`Money` なんて作らず `number` で良い」
**正しい考え方**: 通貨単位や小数誤差、日付範囲の制約などは、プリミティブのままでは表現できない。VO にすることで **不正な状態を型レベルで排除** できる（Primitive Obsession の回避）。

### 誤解③: 「VO は `readonly` を付ければ完成」
**誤り**: 「フィールドに `readonly` を付ければ VO だ」
**正しい考え方**: 不変性に加え、**等価性の判定基準**（`equals()`）と **操作の結果が新しい VO を返す** という2点が揃って VO が成立する。setter を生やした瞬間 VO ではなくなる。

### アンチパターン: Primitive Obsession（原始型中毒）
ドメインの意味を持つ値（金額・住所・識別子など）を `string` や `number` のまま扱う。型の手がかりがなくなり、「商品IDと顧客IDを取り違える」「円とドルを足す」といったバグを実行時まで検知できない。

## 5. 実装上の判断軸

### エンティティを採用すべきサイン
- 同じ属性でも別物として追跡する必要がある（顧客・注文・口座）
- ライフサイクルがある（作成 → 変更 → 削除、状態遷移）
- 履歴や監査の対象になる
- 属性が時間とともに変化しても「同じもの」として扱いたい

### 値オブジェクトを採用すべきサイン
- 属性が等しければ完全に交換可能（金額・住所・日付範囲・座標）
- 不変であるべき（一度決めたら変わらない）
- ID を持たせる意味がない
- 計算や合成の結果として新しい値を作りたい

### 判断に迷ったときのチェックリスト
- [ ] このオブジェクトの 2 インスタンスは、属性が同じなら "同じ" と扱えるか？ → Yes なら VO
- [ ] このオブジェクトは時間経過で属性が変化するか？それでも同一物として扱うか？ → Yes なら Entity
- [ ] 履歴やライフサイクルを追跡する必要があるか？ → Yes なら Entity
- [ ] この値を「合成」「加算」「結合」したくなるか？ → Yes なら VO
