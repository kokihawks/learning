# 集約と集約ルート

## 1. 定義（〜とは）

**集約（Aggregate）** とは、一貫性を保つべき Entity と VO のまとまり。**集約ルート（Aggregate Root）** は集約の唯一の入口となる Entity で、外部からの操作は必ずルートを経由する。集約内部の要素には、外部から直接アクセスしない。

原典: Evans, DDD chapter 6。DDD の戦術的パターンの中でも、整合性設計の中心的概念。

> 一言メタファー: **家族のまとまり**。外部から見れば「〇〇家」という単位で契約や郵便物が届く。家族メンバー個人に外から直接口出ししない。

## 2. なぜ必要か / 解決する問題

### 集約がない場合に起きる問題
- **不変条件が崩れる**: 「注文合計は明細の合計と一致する」「注文明細は 1 つ以上必要」などのルールを複数箇所で守ろうとして、結局どこかで破られる
- **ロック粒度の設計指針がない**: 何をまとめてロックするかが不明瞭で、デッドロックや不整合が起きる
- **並行更新時の衝突**: 複数リクエストが別 Entity を独立に更新し、全体としては矛盾した状態が残る
- **永続化単位が曖昧**: 関連する Entity 群をバラバラに保存して、中途半端な状態が DB に残る

### 集約が解決すること
- **不変条件がルート内でチェックされる**: 集約の内部を守る責務がルートに集中
- **トランザクション境界と一致**: 1 トランザクション = 1 集約 の原則で、ACID を効かせやすい
- **並行制御の単位が明確**: 楽観ロックや悲観ロックを集約単位でかける
- **永続化の単位が明確**: 集約ごとにリポジトリを作れる（→ 次トピック）

## 3. 具体例

### 例①: Eコマース（注文 + 注文明細の古典例）

**悪い例: 集約内の要素を外部から直接触る**

```typescript
class Order {
  public lines: OrderLine[] = [];  // 外部から直接 push できてしまう
  public totalPrice: Money = Money.zero();
}

class OrderLine {
  public quantity: number = 1;  // 外部から書き換えできてしまう
}

// 外部コード（UI 層 など）
order.lines.push(new OrderLine(...));  // Order の不変条件チェックを迂回
order.lines[0].quantity = 999;         // totalPrice と不整合になる
```
→ 「合計は明細の合計と一致する」という不変条件が守られない。

**良い例: ルート経由で操作**

```typescript
class Order {  // 集約ルート
  private lines: OrderLine[] = [];

  constructor(
    readonly id: OrderId,
    readonly customerId: CustomerId,
  ) {}

  addLine(productId: ProductId, quantity: number, unitPrice: Money): void {
    if (quantity <= 0) throw new Error("数量は1以上");
    this.lines.push(new OrderLine(productId, quantity, unitPrice));
  }

  totalPrice(): Money {
    return this.lines.reduce(
      (acc, line) => acc.plus(line.subtotal()),
      Money.zero(),
    );
  }
}

class OrderLine {  // 集約内部。private 相当で外部公開しない
  constructor(
    readonly productId: ProductId,
    readonly quantity: number,
    readonly unitPrice: Money,
  ) {}

  subtotal(): Money {
    return this.unitPrice.times(this.quantity);
  }
}
```

> ポイント: `OrderLine` の追加・変更は必ず `Order` メソッド経由。外部は `order.addLine(...)` だけを知り、明細の内部表現や整合性ルールは Order に閉じる。

### 例②: 集約をまたぐ参照は ID で

**悪い例: 他集約を直接オブジェクト参照**

```typescript
class Order {
  constructor(
    readonly id: OrderId,
    readonly customer: Customer,  // Customer 集約を丸ごと保持
  ) {}
}
```
→ Order を読み込むたびに Customer もロードされ、メモリ肥大・整合性責任の混線が起きる。

**良い例: ID 参照で集約の独立性を保つ**

```typescript
class Order {
  constructor(
    readonly id: OrderId,
    readonly customerId: CustomerId,  // 他集約は ID のみ保持
  ) {}
}
```
→ 各集約は独立にロード・保存でき、境界がはっきりする。

## 4. よくある誤解・アンチパターン

### 誤解①: 「集約は大きいほど便利」
**誤り**: 「関連する Entity を全部 1 集約にすれば整合性が保てる」
**正しい考え方**: 大きすぎる集約は並行処理で競合しやすく、パフォーマンスも悪化する。**守るべき不変条件の最小単位** で集約を切るのが原則。

### 誤解②: 「集約内部の Entity にもリポジトリを作る」
**誤り**: 「`OrderLine` も DB テーブルがあるのでリポジトリを作る」
**正しい考え方**: リポジトリは **集約ルートに対してのみ** 作る。内部 Entity はルート経由で取得・変更する。

### 誤解③: 「集約をまたいでも即時一貫性を保てる」
**誤り**: 「注文と在庫は 1 トランザクションで整合させる」
**正しい考え方**: 集約をまたぐ整合性は **結果整合性（eventually consistent）** で設計するのが DDD の通例。ドメインイベント経由で非同期に整合させる（→ B 層「ドメインイベント」）。

### アンチパターン: God Aggregate（巨大集約）
全 Entity を 1 集約に詰め込み、毎回全体をロード・保存する。ロード/保存コスト爆発、ロック競合で並行性ゼロ、変更の影響範囲が不透明 → 典型的な失敗パターン。

## 5. 実装上の判断軸

### 集約として束ねるべきサイン
- 複数要素で同時に守るべき不変条件がある（「明細の合計 = 注文合計」など）
- 1 つのユースケースで必ず一緒に変更される
- ライフサイクルを共にする（削除時に連鎖的に消える）
- 業務上「一枚の紙（伝票）」として扱われる

### 別集約に分けるべきサイン
- 変更頻度やライフサイクルが異なる
- 別々に取得・更新したい場面が多い
- 結果整合性で十分な関係
- 既に独立の業務概念として確立されている（顧客と注文など）

### 集約の大きさを決めるチェックリスト
- [ ] このまとまり全体で守るべき不変条件を挙げられるか？（ないなら分けて良い）
- [ ] 1 つのユースケースでこれらを必ず同時変更するか？
- [ ] 別トランザクションに分けた場合、業務的に許容できない矛盾が起きるか？
- [ ] 他集約への参照は ID 1 本で済むか？（オブジェクト参照になっていたら境界が曖昧）

## 6. 関連する概念

- **[エンティティと値オブジェクト](エンティティと値オブジェクト.md)**: 集約は Entity と VO の集合として構成される。
- **リポジトリ**: 集約は集約単位で永続化する。リポジトリは集約ルートに対応する。
- **[境界づけられたコンテキスト](../../01_戦略的設計/説明/境界づけられたコンテキスト.md)**: 集約はコンテキスト内部の構造。境界を越えると集約も変わる。
- **ドメインイベント**（B層）: 集約をまたぐ結果整合性の担い手。
