# ドメインサービス

## 1. 定義（〜とは）

**ドメインサービス（Domain Service）** とは、あるドメイン操作が特定の Entity や VO に自然に属さない場合に置く、**ステートレスな操作** を表現するパターン。ドメインの語彙で命名され、ドメイン層に属する。

原典: Evans, DDD chapter 5。Entity / VO に収まらないロジックの受け皿として位置づけられる。

> 一言メタファー: **窓口業務**。特定の口座オブジェクトの中に「送金」を書くのは不自然(どちらの口座に書くか？)。かといって UI 層に書くと送金ルールが散る。窓口は明確な責務を持つが、どの口座にも属さない。

## 2. なぜ必要か / 解決する問題

### ドメインサービスがない場合に起きる問題
- **複数集約にまたがる操作の置き場に困る**: 送金・交換・在庫割り当てを Entity に入れると、どちらかに不自然に偏る
- **アプリケーション層へ漏出**: 困ったロジックをアプリ層（ユースケース調整層）に置いてしまい、ドメインルールが散る
- **貧血モデルの逆側の失敗**: すべてを Entity に押し込もうとして、本質的に「2 者の関係性」で成立するロジックを Entity の中で強引に表現してしまう

### ドメインサービスが解決すること
- **ドメイン語彙のまま操作を置ける場所ができる**: 「送金（Transfer）」「割り当て（Allocation）」「照合（Reconciliation）」などが自然に表現できる
- **ドメイン層に留まる**: アプリ層には逃がさず、ユビキタス言語を保つ
- **ステートレスなので再利用・並行利用が安全**

## 3. 具体例

### 例①: Eコマース（注文割り当てサービス）

注文確定時、在庫集約から商品を確保して注文集約に割り当てる処理。**注文と在庫という 2 つの集約にまたがる**。

**悪い例: Order に押し込む**

```typescript
class Order {
  fulfill(inventory: Inventory): void {
    // Order が Inventory を直接触り始める → Order が肥大化し、
    // 責務が「注文の管理」から「在庫の管理」へ越境する
  }
}
```

**良い例: ドメインサービスとして抽出**

```typescript
class OrderAllocationService {
  allocate(order: Order, inventory: Inventory): AllocationResult {
    const shortages: ProductId[] = [];
    for (const line of order.lines()) {
      if (!inventory.hasAvailable(line.productId, line.quantity)) {
        shortages.push(line.productId);
      }
    }
    if (shortages.length > 0) {
      return AllocationResult.shortage(shortages);
    }
    order.lines().forEach((line) =>
      inventory.reserve(line.productId, line.quantity),
    );
    order.markAllocated();
    return AllocationResult.success();
  }
}
```

> ポイント: 「割り当て（Allocation）」という業務概念が明示的に存在し、注文・在庫どちらにも偏らない。将来、優先顧客向けの割当ポリシー等が入っても、このサービス内で拡張できる。

### 例②: 銀行の送金サービス（2 口座間の古典例）

```typescript
class TransferService {
  execute(source: Account, destination: Account, amount: Money): void {
    if (!source.currency().equals(destination.currency())) {
      throw new Error("currency mismatch（ForeignTransferService を使うこと）");
    }
    source.withdraw(amount);
    destination.deposit(amount);
  }
}
```

→ 「送金」は 2 口座の間に成立する概念で、どちらか一方の `Account` に置くと責務が歪む。為替や手数料が絡むと独立クラスとして育つ素地になる。

## 4. よくある誤解・アンチパターン

### 誤解①: 「`〇〇Service` はすべてドメインサービス」
**誤り**: 「名前が `Service` だからドメイン層」
**正しい考え方**: 名前が同じでも、**アプリケーションサービス（ユースケース調整）** と **ドメインサービス（ドメインルール）** は別物。ドメインサービスはドメイン語彙に限り、トランザクション制御や外部連携はアプリ層。

### 誤解②: 「ロジックが長いから切り出す」
**誤り**: 「Entity のメソッドが長くなったのでサービスに移す」
**正しい考え方**: まず Entity 内にリファクタで収まらないか検討。安易にサービスへ逃がすと、Entity がデータ容器化して **貧血ドメインモデル** になる。

### 誤解③: 「状態を持つドメインサービス」
**誤り**: 「キャッシュや進行状態を保持するサービスを作る」
**正しい考え方**: ドメインサービスは原則 **ステートレス**。状態が必要なら、それは Entity か VO ではないか、と問い直す。

### アンチパターン: Everything-is-a-Service
Entity にあるべき振る舞いをすべて `XxxService` に移した結果、Entity がデータ保持だけのクラスになる。DDD の目的（ビジネスルールをモデルに集約する）から逆行する。

## 5. 実装上の判断軸

### ドメインサービスにすべきサイン
- 2 つ以上の集約にまたがる操作（送金・割り当て・照合）
- どの Entity にも自然に属さない（主語が決められない）
- ドメイン語彙で名前が付く（「割り当てる」「送金する」「照合する」など）
- 本質的にステートレスな計算・判定

### ドメインサービスにすべきでないサイン（= 他に置く）
- 1 つの Entity/集約内で完結する → Entity メソッド
- トランザクション境界・リトライ・ログなど調整ロジック → アプリケーションサービス
- 外部 API 呼び出し・DB アクセス → インフラサービス
- 状態を保持する必要がある → Entity に格上げを検討

### 判断チェックリスト
- [ ] この操作は特定の Entity/VO に自然に属するか？ → Yes ならそちらへ
- [ ] 複数集約にまたがるか？ → Yes ならドメインサービス候補
- [ ] ドメイン語彙で命名できるか？ → No ならアプリ層の関心かも
- [ ] ステートレスに書けるか？ → No なら Entity 化を検討
